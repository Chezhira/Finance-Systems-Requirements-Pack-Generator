# v0.3.0 Intake and Target-System Mapping Plan

Finance systems projects start from different levels of process maturity. Some teams have an SOP or workflow document; others have only process knowledge in people's heads. v0.3.0 should support both paths without assuming an SOP exists.

## Goals

- Let users start from an uploaded SOP, a guided SOP-building flow, or the existing manual requirements intake.
- Keep every extracted, drafted, and mapped field reviewable and editable before requirements pack generation.
- Add target ERP/system selection and deterministic candidate fit-gap mapping using curated repository data.
- Preserve the public-safe, deterministic approach: no external AI/API calls, no live web RAG, and no web scraping.

## Intake Modes

### 1. Upload SOP / workflow document

This mode is for users who already have a current-state SOP, desktop procedure, process note, workflow document, or similar artifact.

Planned flow:

1. User uploads a supported document format.
2. App extracts current-state process details into structured intake fields.
3. User reviews and edits the extracted fields.
4. User selects the target ERP/system.
5. App generates candidate fit-gap mapping with validation notes.
6. User reviews and edits the intake and mapping.
7. App generates the requirements pack.

Extracted fields should include process name, process owner, current tools, source documents, key steps, handoffs, data fields, controls, reports, pain points, exceptions, audit evidence, assumptions, and open questions.

### 2. Build SOP from guided finance process questions

This mode is for users who do not have an SOP yet.

Planned flow:

1. User selects the finance process.
2. App asks guided current-state questions about triggers, inputs, systems, actors, steps, approvals, exceptions, outputs, reports, controls, evidence, pain points, and timing.
3. App produces a current-state SOP draft from the answers.
4. User reviews and edits the SOP draft.
5. User selects the target ERP/system.
6. App generates candidate fit-gap mapping with validation notes.
7. User reviews and edits the intake and mapping.
8. App generates the requirements pack.

The generated SOP draft should be treated as a structured draft, not as a final operating procedure. It should include validation prompts for missing owners, unclear handoffs, missing evidence, unclear approvals, and undefined UAT criteria.

### 3. Continue using manual requirements intake

This mode keeps the current MVP path for users who want to enter structured requirements intake directly.

Planned flow:

1. User selects the finance process.
2. User completes or edits structured intake fields.
3. User optionally selects the target ERP/system for candidate mapping.
4. User reviews and edits generated fit-gap notes.
5. App generates the requirements pack.

Manual intake should remain available even after SOP upload and guided SOP modes are added.

## Target-System Mapping

v0.3.0 should ask the user to select the target ERP or finance system before mapping. The mapping must use curated repository data, not live web retrieval.

Planned curated mapping structure:

- Target system name
- Finance process
- Capability area
- Typical supported pattern
- Configuration dependency
- Data dependency
- Control dependency
- Reporting dependency
- Fit-gap prompt
- Validation note

Candidate mapping output should classify items as likely fit, potential gap, configuration dependent, integration dependent, data dependent, or requires SME validation. These are planning notes only; the app must not claim ERP capabilities are guaranteed.

## Review and Edit Gates

Before pack generation, the user should be able to review and edit:

- Extracted SOP fields
- Guided SOP draft
- Current-state process summary
- Pain points and exceptions
- Data requirements
- Controls and evidence needs
- Target-system fit-gap mapping
- Validation notes and open questions

The requirements pack should only be generated from the reviewed structured intake state.

## Explicit Non-Goals

- No external AI/API calls.
- No live web RAG.
- No ERP vendor website scraping.
- No claim that target-system capabilities are guaranteed.
- No confidential uploaded documents in the public demo.
- No automatic pack generation from unreviewed extraction or mapping.
